# V1 Configuration Guide

This file lists the configuration you need before coding V1.

## Local Assumption

For the first version, VideoHub is on the same physical server as Ops Agent Chat.

Even so, V1 should still use the same execution abstraction:

```text
Ops Agent Chat -> SSH -> 127.0.0.1 -> VideoHub project directory
```

Reason:

```text
The code path remains identical when VideoHub later moves to another server.
Only server.host changes from 127.0.0.1 to a remote IP or domain.
```

## Required Environment Variables

```env
APP_ENV=dev
APP_SECRET_KEY=replace-with-a-long-random-string
JWT_EXPIRE_MINUTES=1440

DATABASE_URL=postgresql+psycopg://opsagent:opsagent_password@postgres:5432/ops_agent_chat

DEEPSEEK_API_KEY=replace-with-your-deepseek-api-key
DEEPSEEK_BASE_URL=https://api.deepseek.com
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-v4-pro
LLM_REASONING_EFFORT=high
LLM_THINKING_ENABLED=true

EMBEDDING_PROVIDER=placeholder
EMBEDDING_MODEL=placeholder

ADMIN_USERNAME=admin
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=change-me-before-running

VIDEOHUB_PROJECT_NAME=VideoHub
VIDEOHUB_DEPLOY_TYPE=docker_compose
VIDEOHUB_WORKDIR=/path/to/videohub
VIDEOHUB_COMPOSE_FILE=docker-compose.yml
VIDEOHUB_HEALTH_URL=http://127.0.0.1:8080/health

VIDEOHUB_SSH_HOST=127.0.0.1
VIDEOHUB_SSH_PORT=22
VIDEOHUB_SSH_USERNAME=opsagent
VIDEOHUB_SSH_KEY_HOST_DIR=./secrets
VIDEOHUB_SSH_KEY_PATH=/run/secrets/videohub_ssh_key
```

Note: `VIDEOHUB_WORKDIR` must be replaced with the real VideoHub project path.

## Trusted `cwd`

The command working directory must come from project configuration:

```text
projects.workdir
```

Do not trust `cwd` generated by the Agent.

Recommended runtime rule:

```text
Agent may generate command and purpose.
Backend chooses cwd from database.
SSHExecutor executes trusted cwd + approved command.
```

## VideoHub Localhost SSH Setup

V1 should use a dedicated local SSH user:

```bash
sudo useradd -m -s /bin/bash opsagent
```

Add the public key from the Ops Agent container or host into:

```text
/home/opsagent/.ssh/authorized_keys
```

If V1 needs Docker diagnosis commands:

```bash
sudo usermod -aG docker opsagent
```

Important: Docker group permission is powerful. It is acceptable for V1 local validation, but production should move toward stricter sudoers or a controlled remote runner.
