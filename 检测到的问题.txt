## 2. P0 发布阻断问题

### P0-02 审批 Hash 没有绑定最终执行内容：已修复

- Action 新增不可变 `resolved_spec_json` 和 `rollback_spec_json`。
- 快照包含运行时、workdir、namespace、Compose 文件、连接目标、主机指纹和注册配方；实际执行只读取该快照。
- Action Hash、审批卡和执行器使用同一份快照，敏感字段对 API 输出脱敏。
- 回归：`test_resolved_spec_is_immutable_and_changes_the_approval_hash`。

### P0-03 未识别的变更能力默认验证成功：已修复

- `_verification_satisfied()` 对未知能力明确返回 `False`，没有专用验证规则时 fail closed。
- Registry 强制每个 change Capability 注册 precheck 和 verifier。
- 回归：`test_unknown_change_verification_fails_closed`。

### P0-04 Docker 注册部署验证不解析容器状态：已修复

- 解析 Compose JSON，检查 `State`、`Health`、`ExitCode` 和 `expected_instances`。
- Kubernetes 注册部署使用 `kubectl rollout status` 验证。
- 回归：`test_registered_docker_deployment_rejects_exited_or_wrong_instance_count`。

### P0-05 已成功的 Action 可以再次执行：已修复

- 使用数据库条件更新将 `ready/approved` 原子抢占为 `executing`。
- succeeded、verified、failed、rolled_back 等终态全部跳过。
- 抢占失败不再回滚同事务内其他 Action 的已记录结果。
- 集成回归会对同一 Action 再次调用 execute，并确认执行器调用次数不增加。

## 3. P1 高优先级问题

### P1-01 配置变更没有旧文件 Hash 时前置检查必然通过：已修复

- 注册配置必须提供 `current_sha256`，或显式声明 `allow_create=true`。
- 执行前再次读取文件并精确比较 SHA-256，不再用实际值填充期望值。
- 回归：`test_registered_config_requires_a_real_precondition`。

### P1-02 Registry 不校验引用和执行器：已修复

- Registry 编译阶段校验 executor、effect、risk、approval mode、runtime、precheck、verifier、rollback 及参数兼容性。
- change Capability 缺少 precheck/verifier 时启动失败。
- 回归：`test_registry_rejects_invalid_executors_and_references`。

### P1-03 并发审批可以重复恢复同一个 Run：已修复

- Approval 使用 `UPDATE ... WHERE decision='pending' RETURNING` 原子决策。
- Run 使用 `UPDATE ... WHERE status='waiting_for_approval' RETURNING` 原子进入恢复队列。
- HTTP 审批接口只入队，只有 Worker 能恢复 LangGraph Run。

### P1-04 LLM、SSH 和验证在 HTTP 请求中同步执行：已修复

- 消息接口只创建 queued Run 并返回 HTTP 202。
- 新增独立 `worker` Compose 服务，通过数据库行锁、lease 和 heartbeat 执行任务。
- `/execute` 兼容接口只返回队列状态，不再调用 Graph。
- 回归：`test_http_execute_compatibility_endpoint_never_runs_the_graph`。

### P1-05 Graph 顶层异常不会统一收尾 Run：已修复

- Worker 在 `process_claimed_run()` 顶层捕获异常，写入 failed、错误码和用户可见终态消息。
- 回归：`test_worker_claim_is_single_owner_and_graph_exception_is_terminal`。

### P1-06 取消不能中断当前调用，晚到结果覆盖 cancelled：已修复

- LLM 等待、SSH 命令和 HTTP 流定期检查取消信号。
- `_persist_result()` 以数据库 cancelled 状态为准，拒绝晚到 completed 结果。
- queued Run 取消后也会生成终态 Agent 消息。
- 回归：`test_late_result_cannot_overwrite_cancelled_run`、`test_model_wait_can_be_cancelled_without_accepting_late_result`、`test_cancelling_a_queued_run_creates_a_terminal_assistant_message`。

### P1-07 Run 完成会把整段回答置信度设为 1.0：已修复

- 置信度按 claim 类型限制上限；通用知识不超过 0.7，无证据事实降级为 inference 且不超过 0.5。
- 不再根据 Run completed 直接赋值 1.0。

### P1-08 整段回答只有一个 Claim 且关联全部 Evidence：已修复

- LLM 输出原子 claims，分别标记 fact、inference、recommendation、general_knowledge 或 gap。
- 只关联 claim 明确引用且属于当前 Run 的 evidence id，去重后持久化。
- 集成回归验证两个 Claim 只产生一个精确 Evidence Link。

### P1-09 Approver 有权限但列表看不到审批：已修复

- 审批列表同时查询 requested user、项目 owner 和 approver 成员。
- 回归：`test_approver_membership_is_visible_in_pending_list`。

### P1-10 生产模式不拒绝危险默认配置：已修复

- production 启动会拒绝短或默认 JWT Secret、默认管理员密码、开发数据库密码、宽松 SSH 和缺失模型密钥。
- 回归：`test_production_settings_reject_unsafe_defaults`。

### P1-11 SSH 默认接受未知主机密钥：已修复

- 默认 `SSH_STRICT_HOST_KEY_CHECKING=true`。
- 严格模式要求已登记指纹或系统 known_hosts；指纹不一致立即失败。
- `AutoAddPolicy` 只保留在用户显式关闭严格模式的开发配置中，production 会拒绝该配置。

### P1-12 后端集成测试没有数据隔离：已修复

- pytest session 创建随机独立 PostgreSQL 数据库，执行 Alembic 后运行测试，结束时终止连接并删除数据库。
- 审批恢复集成测试改为调用真实审批服务，不再直接改审批状态。

## 4. P2 中优先级问题

### P2-01 Logout 不撤销 Token：已修复

- JWT 包含 token version 和 jti；logout 原子增加用户 token version。
- 旧 Token 随后访问受保护接口返回 401。
- 回归：`test_logout_revokes_existing_token`。

### P2-02 登录没有限流或失败锁定：已修复

- 新增 PostgreSQL `login_throttles`，按客户端和登录标识的 Hash 计数。
- 连续 5 次失败后锁定 5 分钟，使用 UPSERT 处理并发失败请求。
- 回归：`test_login_lockout_and_inactive_user_rejection`。

### P2-03 Health Check 只返回静态结果：已修复

- `/live` 只检查 API 进程存活。
- `/ready` 和 `/health` 检查数据库、LangGraph checkpoint、模型配置和最近 Worker 心跳。
- 回归：`test_health_checks_dependencies_and_reports_consistent_version`。

### P2-04 同一项目可以有多个默认环境：已修复

- PostgreSQL 部分唯一索引保证每个项目最多一个 active default。
- API 设置新默认环境时先取消旧默认，并处理 IntegrityError 为 HTTP 409。
- 回归：`test_environment_api_enforces_schema_and_single_default`。

### P2-05 Environment API 长度校验与数据库不一致：已修复

- name、workdir、namespace 等字段加入与数据库一致的 Pydantic 长度约束。
- 配置或运行时切换校验失败统一返回 422，不再落到数据库 500。

### P2-06 Environment 配置缺少分类型 Schema：已修复

- 为 manual、docker_compose、kubernetes、systemd、mixed 定义类型化配置模型。
- 校验 services、健康端点、Compose 路径、Kubernetes manifest、注册部署和注册配置。
- runtime 改变时会重新校验已有 config_json。

### P2-07 HTTP 3xx 默认判定健康：已修复

- 默认成功状态仅为 200 和 204；不跟随重定向。
- 环境可以显式登记其他允许状态，但 302 不再默认健康。
- 回归：`test_http_redirect_is_not_healthy_and_request_uses_pinned_address`。

### P2-08 HTTP 检查存在二次 DNS 解析窗口：已修复

- DNS 只解析一次，请求固定连接已校验 IP，并保留原 Host 与 HTTPS SNI。
- 禁止 link-local、multicast、reserved 和 unspecified 地址。

### P2-09 SFTP 没有验证符号链接最终路径：已修复

- 对 workdir 下每个路径组件使用 `lstat`，拒绝任何符号链接和目录逃逸。
- 回归：`test_sftp_safe_target_rejects_symlink_components`。

### P2-10 配置更新临时文件名固定：已修复

- 临时文件加入随机 UUID；写入失败优先恢复原文件，再处理临时文件清理错误。
- 回归：`test_registered_file_write_failure_restores_original_backup`。

### P2-11 前端核心依赖使用 latest：已修复

- React、Vite、TypeScript、Lucide 及类型包均固定精确版本。
- Dockerfile 使用 `npm ci` 和 lockfile 复现构建。

### P2-12 前端没有自动化测试入口：已修复

- 新增 `npm test` 和 `npm run test:e2e`。
- 覆盖 Run 终态、审批乐观状态、恢复文案和会话切换后的轮询竞争。
- 本轮 4 项状态测试全部通过，生产构建通过。

### P2-13 仓库没有 CI 工作流：已修复

- 新增 `.github/workflows/ci.yml`。
- CI 包含 Registry 编译、迁移往返、后端 pytest、Docker Adapter、前端测试、前端构建、HTTP smoke、Compose 校验和镜像构建。

### P2-14 没有真正的自动回滚闭环：已修复

- 变更执行失败、缺少 verifier 或验证失败都会进入恢复流程。
- 服务恢复基于前置检查保存的原状态；重启前运行则恢复运行，重启前停止则恢复停止。
- 配置恢复使用独立备份并验证旧 Hash；部署恢复后再次检查 Compose/Kubernetes 状态。
- Capability 恢复动作重新经过 Policy；只有原审批 Hash 已包含该恢复方案时才允许自动执行。
- 没有可靠恢复方案时审批卡明确显示“无可自动执行的恢复步骤”，系统不会宣称恢复成功。

### P2-15 自动测试没有覆盖真实 Runtime Adapter：已修复到当前发布范围

- 新增 opt-in 真实 Docker Compose Adapter 测试，启动临时 pgvector 服务并通过 Adapter 读取运行状态，最后销毁资源。
- SSH/SFTP/HTTP 使用确定性传输与安全边界测试；Kubernetes/systemd 尚无真实集群级 CI，仍属于对应运行时接入时的验收项。
- 本轮因 Docker socket 权限未执行真实 Docker 用例，CI 和具备 Docker 权限的本机可执行。

### P2-16 Audit Hash Chain 只有写入没有校验器：已修复

- 新增链结构、事件 Hash、分叉、断链和循环校验器。
- 新增管理员接口 `GET /api/audit-events/verify`。
- append 使用 PostgreSQL advisory transaction lock 串行化链头写入。
- 回归：`test_audit_chain_verifier_detects_and_reports_tampering`。

## 6. 伴随修复

逐项复核时额外修复了以下状态一致性问题：

- read、precheck、verifier Action 执行后进入明确 succeeded/failed 终态。
- Worker lease 过期时，把仍处于 executing 的 Action 标为 `execution_unknown`，不自动重放可能已生效的变更。
- 过期 Worker 的晚到结果不能覆盖系统生成的失败终态。
- 前端轮询结果绑定原会话，切换会话后旧请求不能覆盖新会话消息。
- 多 Action 审批时，批准其中一项不会错误提示“整个变更流程已完成”。
- 移除 Pydantic Core 不支持的正则负向前瞻，注册路径改用 `PurePosixPath + AfterValidator` 校验，避免后端重建后在导入阶段启动失败。
